# 요구사항 문서

## 소개

이 문서는 theory/december 디렉토리에 생성할 2시간 다이어그램 워크샵 자료의 요구사항을 정의합니다. 이 워크샵은 초급 DevOps 엔지니어가 Mermaid와 Draw.io를 사용하여 아키텍처 다이어그램을 작성하고, 이를 통해 시스템을 빠르게 이해하고 실제 구축을 위한 기초 자료를 만드는 방법을 학습하도록 돕습니다.

## 워크샵 개요

- **대상**: 초급 DevOps 엔지니어
- **시간**: 총 2시간 (강의 50분 + 휴식 10분 + 실습 60분)
- **도구**: Mermaid (문서화용), Draw.io (실습용)
- **결과물**: Draw.io로 작성한 다이어그램을 Discord에 제출

## 용어 정의

- **워크샵 시스템**: 2시간 다이어그램 워크샵 교육 자료
- **학생**: 초급 DevOps 엔지니어
- **Mermaid**: 텍스트 기반 다이어그램 도구 (문서화 및 예제용)
- **Draw.io**: 비주얼 다이어그램 에디터 (실습용)
- **아키텍처 다이어그램**: 시스템 구성 요소와 관계를 시각적으로 표현한 것
- **실습 과제**: 학생이 Draw.io로 작성하여 Discord에 제출할 다이어그램

## 요구사항

### 요구사항 1

**사용자 스토리:** 학생으로서, 다이어그램의 중요성과 기본 원칙을 이해하고 싶습니다. 그래야 효과적인 아키텍처 다이어그램을 작성할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 워크샵 자료를 읽을 때 THEN 워크샵 시스템은 다이어그램이 왜 중요한지 설명해야 합니다
2. WHEN 학생이 다이어그램 유형을 배울 때 THEN 워크샵 시스템은 아키텍처, 플로우, 시퀀스 다이어그램의 차이를 설명해야 합니다
3. WHEN 학생이 다이어그램 품질 기준을 배울 때 THEN 워크샵 시스템은 명확성, 일관성, 단순성 원칙을 제공해야 합니다

### 요구사항 2

**사용자 스토리:** 학생으로서, Mermaid 기본 문법을 빠르게 배우고 싶습니다. 그래야 문서에 다이어그램을 포함할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 Mermaid 섹션을 읽을 때 THEN 워크샵 시스템은 순서도, 시퀀스, 아키텍처 다이어그램 문법을 제공해야 합니다
2. WHEN 학생이 Mermaid 예제를 볼 때 THEN 워크샵 시스템은 코드와 렌더링된 이미지를 함께 표시해야 합니다
3. WHEN 학생이 빠른 참조가 필요할 때 THEN 워크샵 시스템은 Mermaid 치트시트를 제공해야 합니다

### 요구사항 3

**사용자 스토리:** 학생으로서, AWS 아키텍처 다이어그램 작성법을 배우고 싶습니다. 그래야 클라우드 인프라를 시각화할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 AWS 섹션을 읽을 때 THEN 워크샵 시스템은 VPC, 서브넷, 보안 그룹 표현 방법을 설명해야 합니다
2. WHEN 학생이 AWS 예제를 볼 때 THEN 워크샵 시스템은 3계층 웹 애플리케이션 다이어그램을 제공해야 합니다
3. WHEN 학생이 AWS 아이콘을 사용할 때 THEN 워크샵 시스템은 공식 AWS 아키텍처 아이콘 위치를 안내해야 합니다

### 요구사항 4

**사용자 스토리:** 학생으로서, Kubernetes 아키텍처 다이어그램 작성법을 배우고 싶습니다. 그래야 컨테이너 오케스트레이션을 시각화할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 Kubernetes 섹션을 읽을 때 THEN 워크샵 시스템은 Pod, Service, Ingress, Deployment 표현 방법을 설명해야 합니다
2. WHEN 학생이 Kubernetes 예제를 볼 때 THEN 워크샵 시스템은 간단한 애플리케이션 배포 다이어그램을 제공해야 합니다
3. WHEN 학생이 Namespace를 표현할 때 THEN 워크샵 시스템은 리소스 그룹화 방법을 설명해야 합니다

### 요구사항 5

**사용자 스토리:** 학생으로서, CI/CD 파이프라인 다이어그램 작성법을 배우고 싶습니다. 그래야 배포 프로세스를 문서화할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 CI/CD 섹션을 읽을 때 THEN 워크샵 시스템은 순서도 기호와 의미를 설명해야 합니다
2. WHEN 학생이 CI/CD 예제를 볼 때 THEN 워크샵 시스템은 빌드-테스트-배포 파이프라인 다이어그램을 제공해야 합니다
3. WHEN 학생이 의사결정 흐름을 표현할 때 THEN 워크샵 시스템은 조건부 분기 표현 방법을 설명해야 합니다

### 요구사항 6

**사용자 스토리:** 학생으로서, 핵심 로직 플로우 다이어그램 작성법을 배우고 싶습니다. 그래야 비즈니스 로직을 시각화할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 로직 플로우 섹션을 읽을 때 THEN 워크샵 시스템은 순차, 조건부, 반복 패턴을 설명해야 합니다
2. WHEN 학생이 로직 플로우 예제를 볼 때 THEN 워크샵 시스템은 사용자 인증 흐름 다이어그램을 제공해야 합니다
3. WHEN 학생이 복잡한 로직을 표현할 때 THEN 워크샵 시스템은 하위 프로세스 분해 방법을 설명해야 합니다

### 요구사항 7

**사용자 스토리:** 학생으로서, Draw.io 사용법을 배우고 싶습니다. 그래야 실습 과제를 완료할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 Draw.io 가이드를 읽을 때 THEN 워크샵 시스템은 기본 사용법과 단축키를 설명해야 합니다
2. WHEN 학생이 AWS 아이콘을 찾을 때 THEN 워크샵 시스템은 Draw.io에서 AWS 라이브러리 추가 방법을 설명해야 합니다
3. WHEN 학생이 다이어그램을 저장할 때 THEN 워크샵 시스템은 파일 형식과 내보내기 옵션을 설명해야 합니다

### 요구사항 8

**사용자 스토리:** 학생으로서, 실습 과제 가이드를 원합니다. 그래야 단계별로 다이어그램을 작성할 수 있습니다.

#### 인수 기준

1. WHEN 학생이 실습 과제를 시작할 때 THEN 워크샵 시스템은 4가지 실습 과제를 제공해야 합니다
2. WHEN 학생이 각 과제를 수행할 때 THEN 워크샵 시스템은 단계별 가이드와 체크리스트를 제공해야 합니다
3. WHEN 학생이 과제를 완료할 때 THEN 워크샵 시스템은 제출 방법과 평가 기준을 설명해야 합니다
4. WHERE 실습 과제가 정의될 때 THEN 워크샵 시스템은 AWS 아키텍처, Kubernetes 배포, CI/CD 파이프라인, 로직 플로우 과제를 포함해야 합니다
