## 👥 팀 구성 원칙

### 팀 크기 및 구성

**권장 팀 크기**: 3-4명
- **3명**: 빠른 의사결정, 명확한 역할
- **4명**: 역할 분담 용이, 백업 가능

**팀 구성 원칙**:
```
다양성 + 상호보완 = 강한 팀

Frontend 개발자 (1명)
Backend 개발자 (1-2명)
DevOps 담당 (1명)
PM/기획 (겸임 가능)
```

### 역할 및 책임

**Frontend 개발자**:
- UI/UX 구현
- API 연동
- 사용자 경험 최적화
- 반응형 디자인

**Backend 개발자**:
- API 설계 및 구현
- 데이터베이스 설계
- 비즈니스 로직
- 보안 및 인증

**DevOps 담당**:
- Docker 컨테이너화
- Kubernetes 배포
- CI/CD 파이프라인
- 모니터링 설정

**PM/기획** (겸임):
- 프로젝트 일정 관리
- 요구사항 정리
- 문서화 총괄
- 발표 준비

### 팀 구성 방법

**Step 1: 자기소개** (5분)
```
이름:
배경: (전공, 경험)
강점: (잘하는 것)
관심사: (하고 싶은 것)
선호 역할:
```

**Step 2: 팀 매칭** (10분)
- 관심사가 비슷한 사람끼리
- 역할이 상호보완되는 사람끼리
- 학습 목표가 유사한 사람끼리

**Step 3: 팀명 정하기** (5분)
- 프로젝트 주제 관련
- 기억하기 쉬운 이름
- 긍정적인 의미

## 💡 주제 선정 가이드

### 좋은 주제의 조건

**SMART 원칙**:
- **S**pecific: 구체적인 목표
- **M**easurable: 측정 가능한 성과
- **A**chievable: 4주 내 달성 가능
- **R**elevant: November 학습 내용 활용
- **T**ime-bound: 명확한 일정

**체크리스트**:
- [ ] 팀원 모두 관심 있는 주제인가?
- [ ] 4주 내 완성 가능한가?
- [ ] November 학습 내용을 활용하는가?
- [ ] 포트폴리오 가치가 있는가?
- [ ] 실생활 문제를 해결하는가?

### 주제 선정 프로세스

**Step 1: 브레인스토밍** (10분)
```
각자 아이디어 3개씩 제시
    ↓
포스트잇에 작성
    ↓
벽에 붙이고 그룹핑
    ↓
비슷한 아이디어 묶기
```

**Step 2: 평가 및 선택** (10분)
```
각 아이디어를 SMART 기준으로 평가
    ↓
점수 매기기 (1-5점)
    ↓
상위 3개 선정
    ↓
팀 투표로 최종 결정
```

**Step 3: 범위 정의** (10분)
```
핵심 기능 정의 (Must Have)
    ↓
추가 기능 정의 (Nice to Have)
    ↓
제외 기능 정의 (Out of Scope)
    ↓
MVP 범위 확정
```

**기술 기반 접근**:

2. 새로운 기술 실험
3. 기술 조합 시도

**트렌드 기반 접근**:
1. 최신 기술 트렌드
2. 인기 서비스 벤치마킹
3. 오픈소스 프로젝트 참고

## 📋 프로젝트 계획 수립

### 요구사항 정의

**기능 목록 작성**:
```markdown
## 필수 기능 (Must Have)
- [ ] 사용자 인증 (회원가입, 로그인)
- [ ] 핵심 기능 1
- [ ] 핵심 기능 2
- [ ] 핵심 기능 3

## 추가 기능 (Nice to Have)
- [ ] 부가 기능 1
- [ ] 부가 기능 2

## 제외 기능 (Out of Scope)
- 복잡한 기능 1
- 시간이 많이 걸리는 기능 2
```

**사용자 스토리**:
```
As a [사용자 유형]
I want to [기능]
So that [목적]

예시:
As a 블로그 사용자
I want to 게시글을 작성하고
So that 내 생각을 공유할 수 있다
```

## 🔧 협업 도구 설정

### Git 워크플로우

**브랜치 전략**:
```
main (프로덕션)
  ↑
develop (개발)
  ↑
feature/기능명 (기능 개발)
```

**커밋 메시지 규칙**:
```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷팅
refactor: 코드 리팩토링
test: 테스트 추가
chore: 빌드 업무 수정

예시:
feat: 사용자 로그인 기능 추가
fix: 회원가입 시 이메일 검증 오류 수정
```

**PR 규칙**:
```
1. 작은 단위로 PR 생성
2. 명확한 제목과 설명
3. 최소 1명 이상 리뷰
4. CI 통과 후 머지
5. 머지 후 브랜치 삭제
```

### 커뮤니케이션

**도구 선택**:
- **Slack/Discord**: 일상 소통
- **GitHub Issues**: 작업 관리
- **GitHub Projects**: 칸반 보드
- **Notion/Confluence**: 문서 공유

**회의 규칙**:
- **Daily Standup**: 매일 15분
- **Weekly Review**: 주 1회 1시간
- **Retrospective**: 프로젝트 종료 후
